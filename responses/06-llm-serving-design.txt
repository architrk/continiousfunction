## Foundation Concept #21: LLM Serving at Scale — Prefill vs Decode, Continuous Batching, and KV-Cache Memory Management

### 1) Recommendation + why this is the optimal next step

**Concept #21: “LLM Serving at Scale (Prefill/Decode, Continuous Batching, Paged KV Cache, Disaggregation)”**

This is the missing bridge between **single-request inference tricks** (#19 Efficient Attention, #20 Speculative Decoding) and **real-world frontier deployment**, where the *actual bottlenecks* are:

* **Multi-user throughput vs latency** (how you keep GPUs busy while meeting SLAs),
* **KV-cache memory growth + fragmentation** (what limits batch size in practice),
* **Prefill–decode interference** (why mixing long prompts with decoding wrecks tail latency),
* and **architectural system choices** (continuous batching, paging, and increasingly, prefill/decode *disaggregation*).

It fits your criteria tightly:

* **2024–2025 frontier:** Disaggregated serving and modern KV-cache memory management are active research topics (e.g., DistServe 2024; vAttention 2024 → ASPLOS 2025). 
* **Educational gap:** Tons of people “know” KV cache exists, but *don’t* understand why batching/scheduling/memory are the real limiting factors in production.
* **Natural continuation of #19–20:** #19 makes attention feasible per token; #20 reduces target-model forward passes; **#21 turns those wins into dollars saved** by maximizing GPU utilization under latency constraints.
* **Practical impact:** Every frontier model provider lives or dies by serving efficiency; open systems like **vLLM/PagedAttention** formalize the pattern. ([arXiv][1])

### 2) Math core (2–3 equations)

Use these as the “spine” of the lesson:

**(A) End-to-end latency decomposition**
[
\text{Latency} \approx \underbrace{\text{TTFT}}*{\text{prompt + first token}} ;+; (T*{\text{out}}-1)\cdot \underbrace{\text{TPOT}}_{\text{time per output token}}
]
This is *the* serving mental model: TTFT is dominated by **prefill**, TPOT by **decode**. DistServe explicitly optimizes these separately. 

**(B) Goodput (what systems actually optimize)**
[
\text{Goodput} ;=; \text{Throughput}\times \Pr\big(\text{TTFT}\le S_{\text{TTFT}} ;\wedge; \text{TPOT}\le S_{\text{TPOT}}\big)
]
DistServe is explicitly framed as **goodput-optimized** serving under TTFT/TPOT SLOs. 

**(C) Paging / fragmentation cost for KV cache**
Let (P) be the KV “page/block” size in tokens. A request with total cached length (T) needs (\lceil T/P\rceil) blocks:
[
\text{KV_mem}(T);\propto;\left\lceil \frac{T}{P}\right\rceil\cdot P
\qquad\Rightarrow\qquad
\text{waste}(T);\propto;\left(\left\lceil \frac{T}{P}\right\rceil\cdot P - T\right)
]
This captures the core idea of **PagedAttention**: allocate KV in fixed-size blocks to avoid catastrophic fragmentation and enable dynamic growth/shrink. ([arXiv][1])

### 3) Frontier model connections (where it shows up)

This concept explains *why* frontier inference stacks look the way they do:

* **PagedAttention / vLLM-style block KV management** is now the default mental model for high-throughput open-source serving: dynamic KV growth, near-zero waste, and better batching under concurrency. ([arXiv][1])
* **Prefill vs decode are fundamentally different regimes**:

  * prefill is “many tokens in parallel” (often compute-bound),
  * decode is “one token per step” (often memory-bandwidth constrained due to KV reads). DistServe highlights this distinction and the interference when co-located. 
* **Disaggregation is a 2024→2025 trend**: DistServe shows large gains by putting prefill and decoding on different GPU pools so each can use the most appropriate parallelism and scheduling. 
* **vAttention (2024, to appear ASPLOS 2025)** reframes KV management: keep *virtual* KV contiguous while allocating physical memory on-demand using CUDA virtual memory APIs—reducing kernel complexity and improving throughput vs paged-kernel approaches. ([arXiv][2])

---

### 4) Missing intuition (what people usually *don’t* get)

These are the “aha” points to teach explicitly:

1. **LLM inference is two different workloads in one request**

* Prefill: big parallel matmuls (high arithmetic intensity).
* Decode: tiny matmuls + *huge* KV-cache reads (bandwidth/latency sensitive).
  Mixing them in the same batch creates tail-latency pain. 

2. **Continuous batching is not “bigger batches” — it’s “one token per request per step”**
   Static batching assumes requests start together. Real traffic doesn’t. Continuous batching keeps a rolling set of active sequences so the GPU stays busy even as requests arrive/finish.

3. **The KV cache isn’t just “memory usage” — it’s a *scheduler constraint***
   Batch size is bounded by KV memory, which grows with generated tokens and varies per request. Naively allocating contiguous buffers wastes memory through fragmentation; paging makes growth cheap and predictable. ([arXiv][1])

4. **Disaggregation is about removing *interference*, not just scaling**
   DistServe’s core claim is that colocating prefill and decode couples resource allocation and harms both TTFT and TPOT under SLOs; separating them lets each phase be optimized independently.

### 5) Canonical papers (2–3) with citations

1. **Efficient Memory Management for Large Language Model Serving with PagedAttention** — *Kwon et al., 2023 (SOSP)* ([arXiv][1])
   The foundational “KV as pages/blocks” idea + vLLM.

2. **DistServe: Disaggregating Prefill and Decoding for Goodput-optimized Large Language Model Serving** — *Zhong et al., 2024 (OSDI)* 
   Makes TTFT/TPOT separation and interference the central systems story.

3. **vAttention: Dynamic Memory Management for Serving LLMs without PagedAttention** — *Prabhu et al., 2024 (arXiv; to appear ASPLOS 2025)* ([arXiv][2])
   A newer direction: leverage CUDA/VM abstractions to avoid rewriting attention kernels for paged layouts.
